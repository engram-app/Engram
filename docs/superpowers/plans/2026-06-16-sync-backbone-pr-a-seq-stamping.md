# Sync Backbone PR A — `seq` columns + per-vault counter + in-transaction stamping Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a monotonic `seq` to every notes/attachments row, allocated from a per-vault `vaults.change_seq` counter inside the same DB transaction as the row write, plus a backfill for existing rows. The timestamp `/changes` feed stays primary; this PR only lays the ordering substrate.

**Architecture:** A single allocator `Engram.Vaults.next_seq!/1` does `UPDATE vaults SET change_seq = change_seq + 1 ... RETURNING change_seq` inside the caller's existing `Repo.with_tenant` transaction (RLS + atomicity for free). Every *leaf* write function (insert/update/soft-delete/bulk) calls it and stamps the row(s); batch wrappers inherit stamping through the leaf functions they call. One `seq` is allocated per write operation; all rows touched by a bulk op share that op's `seq` (ties are fine — keyset pagination by `(seq, id)` is PR B's concern).

**Tech Stack:** Elixir 1.17+, Phoenix 1.8+, Ecto, PostgreSQL (RLS multi-tenancy via `Repo.with_tenant`), ExUnit + ExMachina factories, migration-safety `phase/*` labels + `structure.sql` schema dump.

**Spec:** `docs/superpowers/specs/2026-06-16-sync-change-log-backbone-design.md` (rollout step "PR A").

### Task 1: Migration — add `change_seq` + `seq` columns (phase/expand)

**Files:**

- [ ] **Step 1: Write the migration**

`CREATE INDEX CONCURRENTLY` cannot run inside a transaction, so disable the migration transaction + lock. `change_seq` is `NOT NULL DEFAULT 0` (constant default = no table rewrite in PG11+); `seq` is **nullable** (backfilled in Task 7; a later contract PR can tighten it).

```elixir
defmodule Engram.Repo.Migrations.AddChangeSeqExpand do
  use Ecto.Migration

  @moduledoc """
  Sync change-log backbone, step A. phase/expand: purely additive.
  - vaults.change_seq: per-vault monotonic counter (source of seq values).
  - notes.seq / attachments.seq: latest change sequence per row (nullable
    until backfilled in 20260616120100_backfill_seq).
  CREATE INDEX CONCURRENTLY requires no surrounding transaction.
  """

  @disable_ddl_transaction true
  @disable_migration_lock true

  def up do
    alter table(:vaults) do
      add :change_seq, :bigint, null: false, default: 0
    end

    alter table(:notes) do
      add :seq, :bigint
    end

    alter table(:attachments) do
      add :seq, :bigint
    end

    create index(:notes, [:vault_id, :seq], concurrently: true)
    create index(:attachments, [:vault_id, :seq], concurrently: true)
  end

  def down do
    drop index(:notes, [:vault_id, :seq])
    drop index(:attachments, [:vault_id, :seq])

    alter table(:notes) do
      remove :seq
    end

    alter table(:attachments) do
      remove :seq
    end

    alter table(:vaults) do
      remove :change_seq
    end
  end
end
```

  @doc """
  Allocates the next monotonic change sequence for a vault.

  MUST be called inside the caller's existing `Repo.with_tenant/2`
  transaction so the bump participates in the same atomic unit as the row
  write it stamps (and inherits RLS tenant context). The row-level lock on
  the vault row serializes seq assignment per vault.

  def next_seq!(vault_id) do
    %{rows: [[seq]]} =
      Repo.query!(
        "UPDATE vaults SET change_seq = change_seq + 1 WHERE id = $1 RETURNING change_seq",
        [Ecto.UUID.dump!(vault_id)]
      )

    seq
  end

### Task 7: Backfill existing rows (phase/migrate-data)

**Files:**
- Create: `priv/repo/migrations/20260616120100_backfill_seq.exs`
- Modify: `priv/repo/structure.sql` (regenerated)

> Assigns a deterministic seq to pre-existing rows (`updated_at` order within each vault) and sets each vault's `change_seq` to its current max, so newly allocated seqs continue strictly above the backfilled ones. Runs in batches via raw SQL (RLS bypassed in migration context — migrations run as the owner). Idempotent: only touches rows where `seq IS NULL`.

- [ ] **Step 1: Write the backfill migration**

```elixir
defmodule Engram.Repo.Migrations.BackfillSeq do
  use Ecto.Migration

  @moduledoc """
  Sync change-log backbone, step A backfill. phase/migrate-data.
  Assigns seq to pre-existing notes/attachments per vault in updated_at
  order, then advances each vault.change_seq to its max so future
  next_seq! values stay strictly increasing. Idempotent (seq IS NULL only).
  """

  def up do
    # 1. Number existing notes per vault by (updated_at, id), assign seq.
    execute("""
    WITH numbered AS (
      SELECT id,
             row_number() OVER (PARTITION BY vault_id ORDER BY updated_at, id) AS rn
      FROM notes
      WHERE seq IS NULL
    )
    UPDATE notes n SET seq = numbered.rn
    FROM numbered WHERE n.id = numbered.id
    """)

    # 2. Continue numbering attachments AFTER notes within each vault.
    execute("""
    WITH maxn AS (
      SELECT vault_id, COALESCE(max(seq), 0) AS base FROM notes GROUP BY vault_id
    ),
    numbered AS (
      SELECT a.id,
             COALESCE(m.base, 0)
               + row_number() OVER (PARTITION BY a.vault_id ORDER BY a.updated_at, a.id) AS seq
      FROM attachments a
      LEFT JOIN maxn m ON m.vault_id = a.vault_id
      WHERE a.seq IS NULL
    )
    UPDATE attachments a SET seq = numbered.seq
    FROM numbered WHERE a.id = numbered.id
    """)

    # 3. Advance each vault's counter to the max seq it has handed out.
    execute("""
    UPDATE vaults v SET change_seq = GREATEST(v.change_seq, sub.maxseq)
    FROM (
      SELECT vault_id, max(seq) AS maxseq FROM (
        SELECT vault_id, seq FROM notes
        UNION ALL
        SELECT vault_id, seq FROM attachments
      ) all_rows GROUP BY vault_id
    ) sub
    WHERE v.id = sub.vault_id
    """)
  end

  def down do
    # Reversible enough for pre-launch: clear backfilled values.
    execute("UPDATE notes SET seq = NULL")
    execute("UPDATE attachments SET seq = NULL")
    execute("UPDATE vaults SET change_seq = 0")
  end
end
```
